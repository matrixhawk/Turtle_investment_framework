# 龟龟投资策略 v1.0 — 视频脚本（框架 + 口播）

> 目标时长：~20 分钟 | 两大部分 | 面向：有一定投资基础、对 AI+量化感兴趣的观众
> 语言风格：技术科普（严谨但不枯燥，适度类比）

---

## 开场 (1 min)

**画面**：分析报告截图 + 选股器排名表快速闪过

**口播**：

> 如果让 AI 帮你做一份完整的股票基本面分析，需要几步？
>
> 答案是六步——但不是让 AI 从头到尾一口气干完。
>
> 这个项目叫"龟龟投资策略"，它的核心设计理念是：**能算的让 Python 算，算不了的才让 LLM 判断**。确定性的财务计算交给脚本，不可还原的定性判断交给大模型。
>
> 今天这期视频分两部分：第一部分，我会用 Jupyter Notebook 实操演示选股器——从五千只 A 股里，自动筛选出值得深入研究的标的；第二部分，我会拆解整个 prompt 系统的架构——六个阶段、四个因子、为什么要这样设计。
>
> 我们直接开始。

## Part 1：龟龟选股器 (8-9 min)

### 1.1 选股器是什么？(1 min)

**画面**：打开 `notebooks/screener.ipynb`，展示 Notebook 全貌

**口播**：

> 龟龟选股器做的事情很简单：从全市场五千多只 A 股里，通过两层筛选，找出大概 50-100 只值得你花时间深入分析的股票。
>
> 为什么是两层？因为这模拟了人工选股的自然流程——先用几个硬指标快速过滤掉明显不合格的，再对剩下的逐只做深入财务分析。
>
> 第一层叫 Tier 1，粗筛，五秒钟搞定全市场。第二层叫 Tier 2，深筛，每只股票要调八个 API，花五到十秒做详细分析。
>
> 整个流程在一个 Jupyter Notebook 里就能跑完。我们来看。

### 1.2 Tier 1 粗筛演示 (2.5 min)

**画面**：运行 Cell 5（初始化配置）→ Cell 7（Tier 1 过滤）→ Cell 9（排名结果表格）

**口播**：

> 首先是初始化。这个 Cell 创建了一个 ScreenerConfig 对象，里面定义了所有的过滤阈值和评分权重。我待会儿会讲怎么自定义这些参数。
>
> 现在运行 Tier 1。
>
> 【等待运行完成，~5秒】
>
> 五秒，结束了。我们看看发生了什么。
>
> 【切到漏斗图 → `video/assets/01_tier1_funnel.png`】
>
> Tier 1 总共经过了八道过滤器，像一个漏斗一样：
>
> 起点是全市场大约五千只股票。第一步，剔除 ST 和退市股——这些要么有退市风险，要么已经不适合普通投资者了。剩下大约四千三。
>
> 然后剔除银行股。不是因为银行不好，而是银行的财务结构太特殊，用通用的筛选框架评估不公平。
>
> 接着是上市满三年——太新的公司数据不够，没法做有意义的财务分析。市值大于五亿——太小的流动性差。换手率大于千分之一——过滤掉僵尸股。PB 小于等于十——排除泡沫化严重的。
>
> 到这里还剩大约一千四百只。
>
> 然后是一个关键的分流步骤：**PE 双通道**。盈利的股票（PE 大于零且小于五十、有分红）进入主通道，大约七百只；亏损或者没有 PE 的进入观察通道，按市值取前五十只。观察通道的存在是为了不错过那些暂时亏损但可能有价值的大公司。
>
> 最后对主通道做综合评分排序。公式很简单：四成权重给股息率，三成给市盈率倒数，三成给市净率倒数。本质上就是找便宜且能分红的。
>
> 取主通道前一百五十名加上观察通道五十名，总共两百只股票，进入 Tier 2。
>
> 【切回 Notebook Cell 9 的排名表格】
>
> 这是 Tier 1 的排名结果。注意，这一步只调了两个 API——stock_basic 和 daily_basic，成本极低。

### 1.3 Tier 2 深筛演示 (3 min)

**画面**：运行 Cell 11（tqdm 进度条滚动）→ Cell 13（最终排名 styled 表格）

**口播**：

> 现在进入 Tier 2，这才是真正的深度分析。
>
> 【运行 Cell 11，展示 tqdm 进度条】
>
> 你可以看到进度条在走，每只股票大概五到十秒。因为每只股票要调八个 API：利润表、资产负债表、现金流量表、分红记录、财务指标、审计意见、股权质押、周线行情。
>
> 在等待的过程中，我来讲讲每只股票经历了什么。
>
> 【切到 Tier 2 流水线流程图 → `video/assets/02_tier2_pipeline.png`】
>
> 首先是**硬否决**，也就是一票否决制。两个条件触发直接淘汰：大股东质押比例超过 70%——说明控制权不稳；审计意见不是"标准无保留"——说明财务可能有问题。这两个是红线，碰了就出局。
>
> 通过硬否决的，进入**财务质量门**。三个指标：加权 ROE 大于 8%、毛利率大于 15%、资产负债率低于 70%。不达标的也剔除。
>
> 然后进入核心计算：**穿透回报率**。这个概念是整个龟龟策略的灵魂——不看报表利润，而是追踪实际到手的现金流，算出真实的股东回报率。这个值会和一个门槛值比较，门槛值取无风险利率加三个百分点和百分之五取高者。
>
> 接下来计算一系列**估值指标**：EV/EBITDA、扣除现金后的 PE、自由现金流收益率等等。
>
> 最后算**地板价**——用五种方法分别估一个极端保守的买入价格，取算术平均值，然后计算当前股价相对地板价的溢价率。
>
> 【切到 5 维雷达图 → `video/assets/03_scoring_radar.png`】
>
> 所有指标算完后，用百分位排名法做**五维综合评分**：ROE 占 20%，FCF 收益率 20%，穿透回报率 25%——这是权重最高的，体现了策略的核心理念——EV/EBITDA 占 15%，地板价溢价占 20%。
>
> 【切回 Notebook Cell 13 的最终表格】
>
> 这是最终的排名表。每只股票的五个维度百分位一目了然。综合分越高，意味着这只股票在回报率、估值、安全边际等方面的综合表现越好。

## Part 2：Prompt 系统架构 (9-10 min)

### 2.1 设计哲学 (1.5 min)

**画面**：三个挑战的图示 → `video/assets/04_three_challenges.png` → 混合架构示意图

**口播**：

> 在讲具体架构之前，先回答一个根本问题：**为什么不直接让大模型分析一只股票？**
>
> 你可以试试。给 ChatGPT 或者 Claude 说"帮我分析一下伊利股份"，它会给你一篇像模像样的回答。但有三个致命问题。
>
> 第一，**数据可靠性**。大模型会编数据。它可能告诉你伊利的毛利率是 35%，但实际是 32%。三个百分点的差异，在投资决策里是致命的。所以财务数据必须来自结构化 API，不能让 LLM 去猜。
>
> 第二，**上下文限制**。一家公司五年的财务报表，加上年报附注的关键段落，加上分析规则——随便算算就是十几万字。即使是最大上下文窗口的模型，也可能在分析做到一半时开始丢信息。
>
> 第三，**可复现性**。两个人用同样的 prompt 分析同一只股票，大模型可能给出不同的数字。确定性的数学计算——ROE、现金流折现、安全边际——不应该有随机性。
>
> 所以我们的解决方案是**混合架构**：Python 脚本负责数据采集和确定性计算，LLM 负责那些无法用公式表达的判断——比如"这家公司的护城河是在变宽还是变窄"、"管理层的资本配置能力如何"。
>
> 这就是整个 prompt 系统的设计基础。

---

### 2.2 六阶段流水线总览 (2 min)

**画面**：六阶段流程图 → `video/assets/05_six_phases.png`（动画逐步展开）

**口播**：

> 整个分析流程由一个叫 coordinator.md 的文件调度，它是总指挥。用户输入一个股票代码，coordinator 把任务拆成六个阶段。
>
> 我按执行顺序来讲。
>
> **Phase 0**，PDF 下载。如果用户需要年报附注数据，系统会自动从雪球下载年报 PDF。这一步是条件触发的，可以跳过。
>
> **Phase 1A**，Tushare 数据采集。这是一个 Python 脚本，调用十五个 API，输出一份结构化的 data_pack——里面有五年的利润表、资产负债表、现金流量表，加上股东、分红、质押等等，总共十七个章节。注意，这一步和 Phase 2A 是**可以并行的**。
>
> **Phase 2A**，PDF 预处理。也是 Python 脚本，用关键词匹配从几百页的年报 PDF 里提取七个关键段落：受限资产、应收账龄、关联交易、或有负债、非经常性损益、管理层讨论、子公司。输出一个 JSON 文件。
>
> 到这里为止，全是 Python 脚本，没有 LLM 参与。
>
> **Phase 1B**，WebSearch 补充。Phase 1A 完成后启动，LLM 通过网络搜索补充 API 拿不到的信息——管理层变动、竞争格局、行业动态。
>
> **Phase 2B**，PDF 精析。Phase 2A 完成后启动，LLM 把 Phase 2A 提取的粗文本解析成结构化表格。
>
> **Phase 3**，四因子分析。所有数据到齐后启动，LLM 按照四个因子逐步分析，最终输出一份完整的分析报告。
>
> 关键设计点：每个阶段的 A 和 B 版本是分工明确的——**A 是确定性的 Python，B 是需要判断力的 LLM**。而且 A 完成后 B 可以立即启动，不用等其他阶段。

---

### 2.3 数据采集层详解 (2 min)

**画面**：展示 data_pack_market.md 截图片段 → pdf_sections.json 截图

**口播**：

> 重点讲讲数据采集层，因为这是整个系统的地基。
>
> Phase 1A 的 tushare_collector.py 做了三件事。
>
> 第一，调十五个 Tushare 端点，拉取五年的财务数据。所有金额统一除以一百万，转成"百万元"单位——这是为了和年报的披露习惯一致，也方便 LLM 阅读。
>
> 第二，自动生成第十七章——衍生指标。这是一个很重要的加速设计。比如因子 3 需要计算"真实可支配现金结余"，涉及十几个变量的逐年计算。如果交给 LLM 从原始数据开始算，既慢又容易出错。所以脚本预先把这些中间结果算好，LLM 只需要在此基础上做调整项的加减。
>
> 第三，自动生成数据警告。比如某个字段连续缺失、某个指标同比变动超过 300%、审计意见异常——这些都会被标记出来，告诉 Phase 3 的 LLM "这里要特别注意"。
>
> 【切到 pdf_sections.json 截图】
>
> Phase 2A 的 pdf_preprocessor.py 则是从 PDF 里"钓鱼"。它不读整份年报——那可能有两三百页。它用关键词匹配，精准定位七个关键段落，每段只提取一两千字。
>
> 为什么不让 LLM 直接读 PDF？两个原因：一是 token 成本太高，二是大量无关内容会稀释 LLM 的注意力。先用脚本缩小范围，再让 LLM 在 Phase 2B 做精细解析，效率高得多。

### 2.4 四因子分析体系 (3 min)

**画面**：四因子递进关系图 → `video/assets/06_four_factors.png` → 每个因子的关键概念卡片

**口播**：

> 进入核心——四因子分析体系。四个因子是递进的，像闯关一样，前一个不通过，后面就不用看了。
>
> 【展示因子递进图】
>
> **因子 1：资产质量与商业模式**。分两步。
>
> 第一步叫"五分钟快筛"，六个一票否决项：审计意见异常？审计师频繁更换？有财务造假历史？商业模式说不清楚？没经历过完整经济周期？控股股东有重大红旗？碰到任何一个，直接淘汰。
>
> 通过快筛后进入深度定性分析，九个模块——资本密集度、现金周期、竞争护城河、周期性、人力资本依赖度、管理层质量，等等。这些全是 LLM 做的定性判断，没法用公式表达。
>
> **因子 2：穿透回报率粗算**。这是一个 Top-Down 的快速估算。
>
> 核心概念是 Owner Earnings——巴菲特定义的"企业主盈余"。简单说就是净利润加上折旧摊销，减去维持性资本开支。然后乘以分红比例，除以市值，得到一个粗算的穿透回报率 R。
>
> R 和门槛值 II 比较——II 等于无风险利率加三个百分点和百分之五取高者。如果 R 连 II 的一半都不到，直接否决，不用再做后面的精算了。这一步的意义是快速排除明显不值得深入的标的。
>
> **因子 3：穿透回报率精算**。这是整个系统最重、最核心的部分。
>
> 和因子 2 的 Top-Down 不同，因子 3 是 Bottom-Up——从现金流量表的每一行出发，逐年追踪：真实现金收入是多少？扣掉经营性支出、资本开支、各种调整项，最后能剩下多少"真实可支配现金"？
>
> 这个过程有十一个步骤，用到了年报附注的数据来交叉验证。比如应收账款的账龄分析、关联方交易的占比、或有负债的规模——这些都是 Phase 2 提取出来的数据。
>
> 最终得到一个精算的穿透回报率 GG。默认使用近两年的均值作为基准，因为太早的数据可能被疫情等非经常性事件扭曲。
>
> **因子 4：估值与安全边际**。最后一关。
>
> 安全边际 = GG 减去门槛值 II。如果是正的，说明当前价格买入有超额回报。然后做五项价值陷阱排查——现金流是否在恶化、护城河是否在缩窄、行业是否在衰退、管理层是否在毁灭价值。
>
> 最后反推一个目标买入价，和十年历史价格分位做对比，判断现在的股价处于什么位置。
>
> 四个因子走完，输出一份完整的分析报告。结论只有三种：**买入、观察、排除**。

### 2.5 渐进加载与断点续传 (1.5 min)

**画面**：展示分析报告.md 的增量写入过程（动画/截图对比）

**口播**：

> 最后讲两个重要的工程设计。
>
> 第一个叫**渐进加载**。四个因子的分析规则加起来超过十万字。如果一次性全部塞给 LLM，上下文窗口可能撑不住——要么溢出，要么前面的内容被压缩丢掉。
>
> 所以 Phase 3 采用按需加载：分析因子 1 时，才读取 factor1 的规则文件；因子 1 分析完了，读 factor2 的规则，依次类推。每次只加载当前需要的规则，保证上下文有足够空间。
>
> 第二个叫**断点续传**。每完成一个因子的分析，LLM 会立即把结论写入磁盘上的分析报告文件——不是等全部做完才写，而是增量写入。
>
> 为什么这很重要？因为如果因子 3 做到一半，上下文耗尽了或者出了意外，因子 1 和因子 2 的结论已经安全地存在文件里了。重新启动时可以从断点继续，不用从头来过。
>
> 这两个设计——渐进加载和断点续传——本质上是在对抗 LLM 上下文窗口的物理限制。在这个约束下，你只能把任务拆细、分步执行、中间结果落盘。
